else { cap } is a struct literal. Field shorthand makes any { ident } ambiguous with a block, and it resolved to { cap: cap }, giving "Expected i64 but got { cap: i64 }". I could decode it, but this will bite everyone eventually. Worth a deliberate rule — e.g. struct literal requires : or , or the type being expected, or a parse-time warning when a block position resolves to a shorthand struct.
The allocation-zeroing contract is ambiguous, and the VM broke my assumption of it. Heap paths go through alloc-zeroed/zalloc; arenas rely on fresh mmap. But the VM reuses the tmp arena across #static evaluations without clearing, so my second static eval got dirty slot memory and the probe loop spun forever. I fixed it with an explicit memset (right call regardless), but you should pick one: either "alloc is zeroed" is a real contract the VM upholds, or it's explicitly not a contract and alloc-zeroed's name is misleading. Silent middle ground cost me the worst debugging session of the task.
A #static infinite loop hangs the compiler with zero output. Ten minutes of nothing. A VM step budget that errors out ("static execution exceeded N instructions, last function: ...") would have turned a hang-bisect into an instant diagnosis.
k1 c didn't catch the ICE. The never-body for loop ICE (filed) only fires when IR lowering happens, and c without a #static in the path doesn't lower. A check command that's weaker than the build undermines its value as a fast gate.
Docs teach retired syntax. k1-syntax-basics.md still presents p.x* and reference-through as current; the parser now rejects it. I only knew about .& from prior session notes. The docs drift faster than the language settles — maybe worth a "recently changed" section at the top of the basics doc.
Questions
Should hash live in core next to equals? I put the ability in std/hash.k1, but it's the same kind of foundational thing as equals/print. Relatedly: what's the intended path for hashing user structs/enums? K1 has type schemas + metaprogramming — a meta-derived structural hash (like bitfield.k1 does for bitfields) feels like the natural K1 answer, versus Rust's #[derive]. I'd happily build $derive-hash("my-type") if that's the direction.
Is arithmetic wrapping defined? FNV and fmix64 depend on u64 * wrapping, and it does today in both engines. If overflow ever becomes checked (Rust-debug-style), std hashing breaks. If wrapping is the permanent semantics, fine; if not, K1 needs wrapping operators before that change, and std code should use them now.
Iterator ability: why is peek the primitive? With peek(self: *self) immutable, a filtering iterator like mine can't cache what it found — next() = scan (peek) + scan again (advance-by). Also every nth impl in the codebase is the identical advance-by(n); self.next() — could that be a default? Was peek chosen for for-loop desugaring reasons, or would nextced-as-primitive with peek-via-buffering be on the table?
Is mutation-through-let intended long-term? let m = HashMap/new(); m.insert(...) works because receiver coercion takes the address of the local for a *mut self method. It's ergonomic and I used it happily, but it means let confers no immutability the moment methods exist. Deliberate stance, or placeholder until a let/let mut split?
Literal braces in strings: is {'{'} the blessed spelling? A \{ escape would be kinder for the common case (p-strings make sense for metaprogramming volume, less so for one brace in an assert).
